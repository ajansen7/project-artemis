I'm a builder and a tinkerer. I've been a researcher, an engineer, and a PM — and the through-line has always been a compulsion to understand how things actually work, not just how they're supposed to work.

That instinct is why the API Agents role at OpenAI is the specific role I want to be in right now. Not because "AI is having a moment" — but because I've spent the last two years living inside the exact problem you're trying to solve: developers trying to build reliable agentic systems on top of non-deterministic models, with no real playbook for what "working" even means. I built Smartsheet's LLM evaluation infrastructure from the ground up — a 3-tiered framework (human SME review → LLM-as-a-judge → human-in-the-loop) that became the cross-team standard for when AI features were safe to ship. The hardest part wasn't the tooling. It was convincing engineers that their deterministic QA instincts were actively wrong for generative systems. That conceptual shift — redefining "done" for probabilistic outputs — is something I think about constantly, and it's the exact challenge that every developer building on the OpenAI API is navigating today.

Before I moved into product, I spent seven years as an engineer at Google — building the Pairing Kit Android SDK for Nest's third-party developer ecosystem, engineering the BLE provisioning flow for the Nest × Yale Lock, and working on consumer-scale mobile systems across Nest and Maps. I know what it feels like to be the developer staring at an API reference and wondering why the abstraction is wrong. That fluency is different from developer empathy — it's developer memory. I don't have to imagine the friction; I've felt it.

What I want to do at OpenAI is make it easier for developers to build agents that actually work in production — not just in demos. That means clean, composable primitives that expose the right capabilities without hiding the seams, honest ergonomics for the parts of agentic systems that are still hard (reliability, state management, tool orchestration), and a relentless focus on what developers are actually trying to accomplish rather than what the API can technically do. I think that problem is both extremely hard and extremely important, and I'd love to work on it.
